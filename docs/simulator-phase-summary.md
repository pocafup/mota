## 1. 阶段目标与定位

**项目总目标**（`CLAUDE.md`）：用搜索算法求解魔塔通关后**剩余 HP 最大化**的路径，且架构从一开始就为"复用到任意魔塔"设计。

**模拟器阶段的子目标**（已达成）：

1. 把一座塔的源码（地图 + 事件脚本 + 怪物/道具/门定义）翻译成**确定性状态机**。
2. 以玩家的真实通关存档为金标准，**逐 token 重放并校验**，直到通关。
3. 沉淀一个**塔无关的引擎** + 一套**忠实数据**，为求解器提供可信的"世界模型"和成绩下界。

模拟器是整个项目的地基——求解器再聪明，也只能在模拟器认定的规则里搜索。地基错一处，上层全错。所以本阶段的核心不是"快"，而是"**对**"。

---

## 2. 架构：引擎与塔数据彻底分离

这是 `CLAUDE.md` 的"架构铁律"，也是本阶段贯穿始终的设计：

```
sim/            塔无关的确定性模拟器 + 战斗逻辑（不含任何"第N层如何"的硬编码）
solver/         塔无关的搜索/剪枝（下一阶段）
data/games51/   这座塔特有的一切：51 层地图、怪物、道具、门、地形、事件
extract/        从源码/存档提取并转换成 data/ 的脚本
tests/          单元测试 + 存档重放
docs/           mechanics_51.md（机制事实来源）、handoff.md、solver-design.md
```

- **核心是纯函数 `step(state, action) -> new_state`**：无副作用，便于单测与搜索回溯。
- **数据驱动**：墙是否可通行、怪是否在某格、门用哪把钥匙——全部从 `data/` 读，引擎里没有楼层编号。
- **验证**：50 层做完后，换一座塔 = 换一份 `data/`，引擎与求解器一行不改。本阶段每个机制实现都遵守了这条（例：`getBlockCls`、`switch:`、`footprint` 都是通用分支 + 数据声明，全塔仅个别楼层用到，但实现不绑楼层）。

## 3. 已实现的机制全景

按 `docs/mechanics_51.md` 章节归类（每条都"源码坐实 + 落盘"，非观察推断）：

### 战斗系统（§A）
- 复刻引擎 `getEnemyInfo` + `getDamageInfo` 全链，**8 组 oracle 逐一比对通过**——绝不手写战斗公式。
- 全套 special 修正：先攻(1)、魔攻(2)、坚固(3)、2/3/n 连击(4/5/6)、破甲(7)、反击(8)、净化(9)、模仿(10)、吸血(11)、仇恨(17)、固伤(22)、无敌(20)。

### 属性 / 比率 / 圣水 / 升级（§B）
- ratio 分区（MT1–10=1 … MT41–50=5）；道具拾取×ratio、背包使用不×ratio。
- 圣水：`HP += round(0.74×(atk+def))×10`（JS `Math.round` = `floor(x+0.5)`）。
- 升级：击杀 20 只 → ATK+10 / DEF+10。

### 地形 / 区域伤（§C）
- 区域伤：领域(15)、阻击(18)、夹击(16)、激光(24)、伏击(27) 按格损血；血网/毒地踩格扣血。统一走 `_apply_zone_damage`。

### 商店 / 祭坛 / 钥匙回收（§D）
- 祭坛公共事件"商店"：费用 `20+10*(times1+1)*times1`，**times1 全局累计**跨所有祭坛；HP/ATK/DEF 三档 + 金币守卫（钱不够不成交但仍消化 CHOICE）。
- 钥匙回收商人、商人(trader)≠祭坛(blueShop) 两套独立脚本。

### 后战斗效果（§F）
- 中毒(12，每步−10)、衰弱(13，ATK/DEF−20)、诅咒(14，战后金/经验归零)、自爆(19，HP→1)。

### 飞行系统（§I）
- fly 魔杖（220 次主路径，gate1 连通检查 + canFlyTo/From）、centerFly（中心对称瞬移不换层）、upFly/downFly（±1 层飞翼，硬编码 index 边界，唯一进 MT44 隐藏层的手段）。

### 剧情 / 终局（§M / §N）
- **强制战斗 force**（全塔仅 MT32/MT40 骑士队长，canBattle 拦截的唯一例外）。
- **MT40 "红门上存活怪才打"**（已清格跳过、零伤）。
- **终局 boss 链**：MT49 竞技场 → MT24 传送 MT50 → MT50 终局魔王 → `win`。

### 道具机制（§K / §L）
- 冰魔法 snow（清相邻岩浆）、矿镐 pickaxe（破相邻墙）、地震 earthquake（清当前层全图可破墙）、bomb（炸相邻弱怪，走 afterBattle 路径）。
- 快捷键派发表（49→pickaxe / 50→bomb / 51→centerFly / 52→earthquake）源码坐实，落 `replay_keybindings.json`。

### 专题机关（§G / §G7 / §H）
- MT10 第十层埋伏（踩触发格 → 内层机关门关 → 8 只清完才开）。
- MT3 伏击（强制卸装，永久改变基线 HP400/ATK10/DEF10）。
- MT41 撞假墙揭示链 G7（杀左侧 redWizard 置 flag → 右侧撞假墙放怪 → 杀怪 afterBattle 地形重塑放 downFly）。

### 跨切面机制
- **动态建图**：可达性每步从当前 state 实时计算（看墙/门 flag/钥匙/活怪），事件改图后绝不复用旧可达性。
- **自动模式 floodfill**：help 键开启后，每次同层移动结算后自动秒杀可达零伤怪 + 拾取可达道具，迭代到不动点。
- **双终止态**（见下节）。

## 4. 双终止态：dead / won

这是本阶段为求解器准备的**成功/失败判定接口**，二者对偶——置位后 `step()` 对一切后续 token 冻结 no-op：

| 终止态 | 含义 | 触发 | 对求解器的意义 |
|--------|------|------|----------------|
| **`dead`** | 死亡硬终止 | HP≤0（force 战斗 / 事件扣血 / 地形）| **剪枝**：探索"会死路线"时即时砍掉 |
| **`won`** | 通关软终止 | 杀 MT50(6,5) 魔王 → `afterBattle` 抵达 `type:win` | **成功终点**：此刻 `hero.hp` = 优化目标值 |

> 普通走格战斗由 `canBattle`（damage≥hp 拦截）保证永不致死；能致死的只有 force/事件/地形——这是 `dead` 只在这些来源兜底的依据。

## 5. 关键技术攻坚回顾

本阶段最具代表性的几个难点（均遵守"玩家裁定根因、不自行猜、不凑绿"工作流）：

1. **检查点口径 `tokens[:N+1]`**——所有"差一格/晚一步"偏差的总根因。`tokens[0]=CHOICE:1` 是不计步的初始化事件。
2. **动态可达性**——埋伏机关门、撞墙开路、事件移怪后，旧的静态图必须失效。
3. **多格 boss footprint**——魔龙/章鱼九宫格占位，活着时九格 noPass。
4. **MT40 "存活才打"**——纠正了把 13 连战误展开为无条件战斗的 HP 崖 bug。
5. **MT44 隐藏层 `is_hide`**——`_copy_state` 漏传字段导致每步被重置的根因 bug。
6. **封板的最后一公里（本阶段收尾三连）**：
   - **圣水 no-op**：`_eval_single` 缺 `switch:` 读分支 → MT16 老人二段事件永远到不了发圣水的 choices 支 → tok6340 缺 +6510 HP。
   - **win 永不触发**：`afterBattle["6,5"]` 榜单 while 的 break 被旧码当"丢弃整个剩余指令队列"，连队列末尾的 `type:win` 一起吞掉。修复：break 时只丢弃 `_while_continue` 哨兵、续跑循环外指令。
   - **MT2 商人 3% 祝福**：`_eval_value_expr` 未翻译 JS `Math.round` → eval 异常返回 0（祝福 +0）。

## 6. 验证方法论（铁律的工程化）

- **三道独立验证**：①引擎 oracle 比对战斗损血；②玩家存档逐 token 重放（46 检查点）；③（下一阶段）求解器输出丢回引擎重放。**引擎只当裁判，不进搜索循环。**
- **玩家裁定工作流**：首个 FAIL → 自动把 `[上一PASS → FAIL]` 窗口逐 token 全状态账写入 `checkpoint_ledger.txt` → **交玩家裁定分叉点根因**。绝不自行猜根因、改真值、改断言凑绿。
- **零回归门槛**：68 单测 + 全部 PASS 检查点，每次改动后必须保持。
- **真值订正全部由玩家主动裁定**（如 tok6221→6222、tok4582 楼层、tok5156 金币 oracle），均附依据，非凑绿。

## 8. 下一阶段：求解器

- **目标**：搜索剩余 HP 最大化的通关路径，结果**不得差于基准 14382**。
- **现状**：`docs/solver-design.md` 目前仅 1 条待办（FLY-CHEAP：fly 视为近零代价动作），基本从零起步。
- **已积累的可复用资产**：
  - **自动 floodfill** = 天然的**状态化简器**（自动消化可达零伤怪/道具）。
  - **dead 硬终止** = 现成的**剪枝**接口。
  - **被动加成**（coin 金币×2、knife 屠龙刀×2 魔龙、cross×2 僵尸）= 非线性属性阈值的来源。
  - **centerFly 绕后清场** = MT40 零伤打法范式。
- **设计要点**（`CLAUDE.md` 已警示）：属性边际价值是**非线性/阶梯状**的（防御跨过怪攻、攻击跨过连击/坚固阈值会让损血突变），搜索绝不能假设"每点属性等价"；建图必须动态；玩家存档只作**下界基准**，绝不让求解器模仿其走法。
