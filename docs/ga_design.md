# 遗传算法（GA）搜索范式设计

**写于**：2026-06-13
**作者动机**：beam 搜索在 MT10 红钥/门长臂上结构性卡死（延迟满足），加 4× 预算也推不过去（已实测坐实，见 `docs/handoff.md` §S3 / §🎯钥匙价值 Stage-2）。本文设计一条**专攻延迟满足长臂**的新搜索范式：遗传算法。
**本文档状态**：纯设计。零实现代码、零搜索运行。落地由后续 session 按本文执行。

> **三条红线（贯穿全文，实现期守死）**
> 1. **基因在目标层、不在动作层**（钉死点 1）。
> 2. **潜力 fitness 是对"已跑完死路线"的终评、不构成 κ=1**（钉死点 3）。

## 0. 为什么 beam 卡死、GA 为什么能破

**beam 卡死的精确机制**（`docs/handoff.md` 多个 session 反复坐实）：
- beam 的排序键 `V = HP − Σcost − λ·区势能 + β·引导`（`solver/beam.py:117` `score_points`）是**逐步打分 + 逐 wave 贪心截断**（`solver/quotient.py:576` 每 wave 截到 `beam_k`）。
- 红钥过 boss 是一条**长臂延迟满足**：须先走到红钥 → 花钥匙开 boss 门（**降分：失一把钥匙、穿谷损血**）→ 才够到门后的过 boss 巨值。
- beam 在"降分穿谷"那一段就把这些态截掉了（贪心结构性奖励"每步不亏"、惩罚"先亏后赚"）。`docs/handoff.md` §🎯延迟满足：MT1 道具区净值是正的（穿三怪 `0→−14→−38→−52`、踩宝石 `+21278`），但 beam 走不过中间那道谷。
- **展开层还有硬门控**（`solver/quotient.py:246` `_boundary_ops`）：`keys==0` 时门不发算子、是死边界，逐步解钥的态根本生成不出"门后"的子态。

**GA 为什么能破**：GA 不做逐步贪心剪枝。它在**目标序列空间**里搜索，**一整条路线先跑到底、再按终局评分**。"花红钥匙开门、降分穿谷"这种中途亏损，在 GA 里不会被剪——因为评分只看终局家底，不看中途每一步是否亏。一条"去拿红钥匙 → 开 boss 门 → 打 boss"的基因，解码跑完、终局 HP/潜力高，就当选父母繁殖。**谷被整条跨过，而不是在谷底被截断。**

## 钉死点 1：基因编码 = 目标层，不是动作层

### 1.1 基因定义

一条基因 = **一个有序的"目标"序列**：

```
chromosome = [ goal_1, goal_2, …, goal_n ]
```

每个 `goal` 是一个**兴趣点 + 该兴趣点的离散决策**，类型如下（全部从塔数据涌现，见 1.3）：

| 目标类型 | 含义 | 决策位 |
|---|---|---|
| `pickup_big`  | 去拿某攻防大件（剑/盾级） | 无（拿就是拿） |
| `pickup_gem`  | 去拿某攻防小宝石 | 无 |
| `pickup_key`  | 去拿某把钥匙（含红钥） | 无 |
| `open_door`   | 去开某扇门（花对应钥匙） | 无（开就是开） |
| `trade`       | 去某商人处 | **买 / 不买**（1 bit） |
| `beat_boss`   | 去打某区 boss（通关必经） | 无 |

> **关键**：`open_door` 和 `pickup_key` 必须是**一等目标**。beam 卡死的整个症结就是"花红钥匙开 boss 门"这条长臂——如果基因不能显式表达"去拿这把红钥匙、去开这扇门"，GA 就和 beam 一样到不了门后。这是本范式相对 beam 的核心增量。

### 1.2 论证：为什么不能编码原始动作序列

把基因定义成 `[U, D, L, R, ITEM:n, …]` 的原始动作串，会因为**魔塔的合法动作空间是强约束、稀疏**而让 GA 退化失效。逐条：

1. **随机变异一步几乎必废后续**：把第 50 步 `U` 变异成 `R`，若 `R` 方向是墙/打不动的怪/没钥匙的门，`step` 是 no-op（位置不变，见 `sim/simulator.py:1001` 撞墙分支、`:1192` `damage>=hp` 不战分支、`:1055` 无钥匙门分支）——英雄停在原地，但后续 200 步是为"变异后本应到达的新位置"设计的，全部错位失效。一次单点变异 → 后代基本全废。
2. **交叉必废**：在第 k 位拼接两条动作串，前缀把英雄带到位置 A、后缀是为位置 B 设计的，拼出来后缀立刻撞墙/错位 → 后代几乎全是"前半有效、后半原地打转"的废串。
3. **绝大多数随机串早死或空转**：合法且有意义的动作串在所有动作串里占比极低（要恰好穿对门、打对怪、踩对宝石）。随机采样/变异落在这个稀疏可行集里的概率近 0 → 种群多样性全是噪声、选择压力无从施力。这是 GA 直接编码移动序列解迷宫类问题的**经典死法**。

**目标层编码把"怎么走到目标"externalize 给 decoder**（由确定性引擎保证合法，见钉死点 2）。基因只管"去哪、什么顺序、买不买"——变异/交叉作用在**语义单元（目标）**上：交换两个目标的顺序、插入/删除一个目标、翻转一个商人的买位，得到的后代**仍然是"一串要去的地方"**，永远能解码成一条引擎合法路线（够不到的目标自然被跳过，见 2.3）。搜索空间从"天文数字的动作串、可行集近 0"压缩到"目标的排列组合、几乎处处可行"。

## 钉死点 2：decoder = 现成确定性模拟器

### 2.1 契约

```
decode(chromosome, start_state) -> (action_tokens, final_state)
```

- `start_state`：区/塔的起点态（由现成 `build_start()` 重放得到，与 beam 同源）。
- 用注入的 `sim.simulator.step`（`sim/simulator.py:695`）逐 token 真实推进——它是**纯函数**（首行 `_copy_state` 深拷、无副作用 `:714`）、**确定性**、**可重放**、自带**判死**（`dead`，HP≤0 冻结 `:708`）和**判通关**（`won`，杀魔王 type:win 冻结 `:1528`）。
- 输出的 `action_tokens` 是一条 U/D/L/R/ITEM/FLOOR/CHOICE 串，**保证引擎合法**（因为全程是真 `step` 推进出来的）——可直接丢回真引擎 / `.h5route` 让玩家网站回放（`[[feedback-route-h5route]]`）。

### 2.2 "走向下一个目标"算法（复用现有原语）

decoder 的主循环 = 对基因里每个目标 `g`，从当前态**定向导航**到 `g`：

```
state = start_state
for g in chromosome:
    if 已 won 或 dead: break
    state = navigate_to(state, g)          # 见下；够不到则原样返回
    # 商人目标：到达后按决策位发 CHOICE
    if g.type == trade and g.buy:
        state = step(state, "CHOICE:0")    # 0=买；不买发 CHOICE:1（_resolve_choices 同口径）
# 收尾：能通关则继续奔 boss；否则就地结束
```

`navigate_to(state, g)` 复用 `solver/quotient.py` 的**全套原语**（它们已是塔无关、引擎合法）：

- `_free_cells(state)`（`quotient.py:114`）：从英雄格 floodfill 出**零损血自由块**（门不穿、怪不穿）。
- `_bfs_moves(state, free, target)`（`quotient.py:160`）：块内 BFS 出到 `target` 的 U/D/L/R 串。
- `_boundary_ops(state, free, cross_floor=True)`（`quotient.py:217`）：枚举块边界上的付代价算子——`stair`（上下楼梯，跨层免费边）/ `kill`（杀怪）/ `door`（持钥匙才发）/ `autoopen`（假墙）/ `trigger`（事件格）。
- `_expand_op(state, free, op, step)`（`quotient.py:255`）：把一个算子展开成动作串并真实 `step` 推进。
- `_absorb(state, step)`（`quotient.py:284`）：吸光当前块内所有道具（顺序无关、零损血）。
- `_resolve_choices`（`quotient.py:309`）：解开商人/祭坛 CHOICE 拦截态（真 `step` 发 `CHOICE:i`）。

定向部分（**唯一要新写的核心件**）：
- 若 `g` 的格子已在当前 `free` 块内 → `_bfs_moves` 直接走过去、`_absorb` 吸沿途。
- 否则在 `_boundary_ops` 里挑一个**朝 `g` 推进**的算子展开（开门/杀怪/上楼梯跨块），重算 `free`，循环逼近。"朝 `g` 推进"用 `vzone._toll_dist_from`（`extract/vzone.py:351`，跨层 toll-Dijkstra）做启发：选展开后离 `g` 的 toll-距离最小的算子。
- 没有任何算子能缩短到 `g` 的距离（无钥匙开门 / 打不过拦路怪 / 真不可达）→ **该目标够不到，跳过**（见 2.3）。

> **诚实标注（实现期最大缺口）**：现有 `search_quotient`（`quotient.py:426`）是**无方向 BFS 全展开**，而 decoder 要的是**定向单路推进**（朝指定 `g` 走一条路、不展开兄弟）。所以 `navigate_to` 必须新写，但它 **100% 用上面的现成原语组装、零塔特定硬编码**（楼层/怪/门全由注入 state 的通用字段读出）。`_toll_dist_from` 是"乐观穿门"距离（门/楼梯零代价），只用来**给算子排序选方向**；真正的"够不够得到"由 `navigate_to` 用真 `step` 推进时自然裁定（没钥匙 `door` 算子根本不生成）——乐观距离选错方向最多让这条基因解码得差一点、被 fitness 淘汰，不会产出非法路线。

### 2.3 够不到 / 死亡的处理

- **够不到某目标**（无钥/打不过/不可达）：跳过它，继续基因的下一个目标。基因是"愿望清单"，decoder 尽力满足、做不到就略过——这让基因永远可解码，不会因一个不可行目标整条作废。
- **中途死亡**（`step` 后 `state.dead`）：解码立即终止，终态就是死亡态 → fitness 给死亡低分（见钉点 3）。
- **合法性保证**：因为每个 token 都经真 `step`，撞墙/无钥/打不过的动作要么不被生成、要么是 no-op，**解码产物必然是引擎可重放的合法路线**。

## 钉死点 3：fitness = 真引擎终局局面价值（带"潜力"）

### 3.1 玩家定义的潜力

不只终态 HP，要带"潜力"。对照（`docs/handoff.md` §S3 第 7 条）：α_big=1 那条终 HP 718 但**家底刨光**、潜力枯竭；α_big=0.7 那条终 HP 低但**余血瓶 + 余钥匙 + 更好攻防底子** = 潜力高、最接近人手、已能不靠商人过一区。fitness 必须能把后者评得更高，否则 GA 会选 718 那条耗尽资源的。

潜力 = 一区跑完时：**手里 HP + 地上够得到的剩余血瓶 + 余钥匙 + 攻防底子**。

### 3.2 塔无关量化（关键约束：不能"MT几有什么"硬编码）

每一项都复用**已有的塔无关口径**，全部"像 ΔRP 那样通用"——只认引擎真损血 / items.json pickup 效果 / 门→钥匙映射，不认任何楼层号、怪 id、物品名：

| 潜力项 | 塔无关量化 | 复用的现成件 |
|---|---|---|
| 手里 HP | `state.hero.hp` 直接 | — |
| 攻防底子 | **当前属性对一区剩余存活怪的总减伤压制力** = `−区势能`（区势能=剩余怪 toll 之和，越小=压制越强） | `solver/beam.py:296` `_future_potential` + `build_future_roster`（`:210`）。读活体、引擎真损血、时序性、塔无关 |
| 地上剩余血瓶 | 地图上未拾取的 HP 消耗品总回血量（"留在地上≈银行里的 HP"） | `solver/search.py:93` `_remaining_items`，按 items.json `pickup` 效果数据驱动判 hp 类，**不写死 id** |
| 余钥匙 | 钥匙价值（见下，第一版可简化） | `value_vector` 的 `key:*`（`search.py:60`）；进阶用 `extract/door_value.py` 门后 pocket 可兑现价值口径 |

**推荐的标量 fitness**（最大化复用现成、塔无关）：

```
fitness(final_state) =
      equiv_hp_over_roster(final_state, roster, big, future)   # ← HP + 攻防对剩余怪压制，已塔无关综合
    + w_potion · Σ(地上血瓶回血量)                              # ← _remaining_items
    + w_key    · 余钥匙价值                                     # ← 见 3.4
    + 通关奖励 / − 死亡·未通关惩罚                               # ← 见 3.3
```

- `equiv_hp_over_roster`（`solver/beam.py:139`）已把"HP − Σ对参照怪集的损血 − λ·区势能"综合成一个标量，**正好同时编码了"手里 HP"和"攻防底子"**，且是 beam/重放同一套引擎口径（fitness 与搜索同币种）。
- `roster/big` 用 `region_reference`（`beam.py:125`）或 `build_future_roster` 构一个**固定参照怪集**（GA 无"批"概念，用全塔静态怪位的固定参照，使不同基因的 fitness 可比）。
- `_remaining_items` 现仅扫 `current_floor`——GA 要算**整区/全塔**地上血瓶，需把它的扫描范围扩到区内各层（小改、仍数据驱动）。标为实现期小扩展。

> 权重 `w_potion / w_key / λ / 通关奖励` 待标定。标定原则：让 fitness 在已知的两条 route（α_big=1 的 718 耗尽 vs 0.7 的高潜力）上**把 0.7 评得更高**——用现成 route 当标尺校准（同 `idea3` 用玩家路线反推参数的方法论）。**玩家真实游戏终审是最终裁判。**

### 3.3 死路 / 没通关给低分

- 死亡（`final_state.dead`）：fitness = 大负数 + 小幅"走了多深"奖励（鼓励"虽死但推进更远"的基因，给 GA 爬坡梯度），但恒低于任何活着的基因。
- 没通关但活着：用 3.2 的局面价值（含区势能=离 boss 还差多少压制力，天然给"更接近过关"的态更高分）。
- 通关（`final_state.won`）：大额通关奖励 + 终局 HP（此刻 `hero.hp` 就是项目最终优化目标"剩余 HP 最大化"）。

### 3.4 严密论证：潜力进 fitness 为何**不**构成 κ=1

**κ=1 病的精确机制**（`[[feedback-realized-value]]`、`docs/handoff.md` 多处）：在 beam 的**逐步打分**里给"**还没拿到的**潜力"加分 → 某个守在门口/宝石旁不动的态因"门后/旁边有潜力"得高分 → beam 每一 wave 都选它继续展开 → 英雄 hover 刷分**赖着不动、永不去拿**。

κ=1 成立的三个必要条件：
1. 打分参与**中途逐步决策**（beam 每 wave 用它选留哪些态继续展开）；
2. 加分对象是**未兑现的潜力**（守着就给分、不拿也得分）；
3. 存在"赖着不动也持续得分"的态、能被反复选中。

**GA 的潜力 fitness 一个都不满足**：

1. **没有"中途逐步决策"这回事**。GA fitness 只在**一条完整路线已经跑到底、走法定死之后**，对**终态**评**一次**分。路线已经固定、不再有"下一步留哪个态继续展开"的选择——decoder 推进时**根本不看 fitness**，fitness 不反馈给 decoder 的任何中途决策。条件 1 不成立。
2. **潜力在这里是既成事实、不是悬而未决的期权**。地上血瓶是这条路线**结束时真实留在地上的**、攻防是这条路线**真实攒到的**、余钥匙是**真实剩在手里的**——全是"已经发生"的终局家底，不是"未来可能拿到"的潜力。条件 2 不成立。
3. **繁殖压力反而惩罚"赖着不动"**。fitness 高只让这条基因**更可能当父母繁殖**，不让任何英雄在任何一步赖着不动。一条"原地刷潜力分"的基因解码出来就是"没去成任何目标"→ 终局家底反而差 → 低 fitness → 被淘汰。条件 3 反向成立（赖着不动 = 低分淘汰）。

**一句话**：κ=1 是"打分诱导**单步惰性**"（守着比拿到分高 → 不拿）；GA 的潜力分作用在"**整条已固定路线的繁殖资格**"上，路线无单步可惰，繁殖压力奖励的恰恰是"**真去拿到了高家底**"的路线。英雄是 decoder 里**定向奔向目标**的执行体，不会为 fitness 分神——这与"在 beam 排序键里给未兑现潜力加分"有本质区别。

> 与现有红线一致：beam 侧所有引导项（`pull_big`/`door_pull`/`pickup_bonus`）守的是"只进排序键、拿到才兑现、不进 value_vector"（`extract/big_item_pull.py:18`）。GA 侧守的是"潜力只进**终评**、不进 decoder 的中途推进决策"——**同一红线在两种范式下的对应形态**。

## 钉死点 4：最小验证子问题

**第一版绝不上全塔。** 先用"一区（到 MT10 前）"验证范式本身能不能工作：

- **目标集**：一区的攻防大件（`detect_big_items` 涌现 = 铁剑@MT5 / 铁盾@MT9）+ 攻防小宝石（`detect_big_items` 的 `ranked` 非 big 项）+ 商人（`shops.json` 一区内的）。**第一版可暂不含红钥/boss 门**（那是放大后的长臂主验证）。
- **基因**：这些目标的顺序 + 各商人的买/不买位。
- **验证标准**：GA 能否搜出 **≥ 现 α_big=0.7 那条骨架**——即终局**余潜力高**（余血瓶 + 余钥匙 + 攻防底子 ≥ 0.7 那条）、且**不靠商人或最优地用商人**过一区。能复现/超过 0.7 骨架 = 范式成立。
- **跑通再谈放大**：第二阶段才把 `pickup_key`（红钥）+ `open_door`（boss 门）+ `beat_boss` 接进目标集，验证 GA 能否跨过 beam 跨不过的红钥长臂——**那才是 GA 存在的理由**。

为什么先做这个最小集：它**不含延迟满足长臂**，能在低风险下先把 decoder（navigate_to 定向导航）、fitness（潜力标定）、GA 算子三个新件**各自调通**，再叠加长臂这个难点。一次只引入一类新风险。

## 超参与算子（定义在目标层）

- **基因表示**：变长有序目标序列（排列 + 可选项的子集 + 商人买位）。
- **种群规模 / 代数**：起步 population≈50–100、generations≈50–100（decoder 每次跑全程 `step`，开销见风险节，按实测调）。
- **选择**：锦标赛选择（tournament k≈3）+ **精英保留**（每代留 top-1~2 不变，防最优解丢失）。
- **交叉**：因基因含"顺序"语义，用**顺序交叉 OX / PMX**（保持目标不重复、顺序部分继承）；商人买位按位独立交叉。
- **变异**（全在目标层，保证后代仍可解码）：
  - 交换两个目标的顺序（swap）；
  - 插入一个当前未在基因里的目标 / 删除一个目标（调节"去拿哪些"）；
  - 翻转某商人的买/不买位。
- **fitness 缓存**：decoder 跑全程 `step` 很贵 → 对**相同基因**（规整后）缓存 fitness，去重重复个体。
- **可复现**：固定随机种子，结果可重放（CLAUDE.md 验证铁律：输出路线丢回真引擎重放必须走通）。

## 与 beam 的关系：互补，不取代

- **beam 擅长**：段内 / 短臂的**密集 Pareto 采样**（缩点 + 保护维 + 区势能基分），在"无长臂延迟满足"的区段产出很强（一区剑盾误判已被 `pull_big` 治好，到 MT10 入口 HP 高）。
- **GA 擅长**：**长臂延迟满足 / 目标顺序的全局规划**（红钥过 boss、跨区目标排序）——beam 的逐步贪心剪枝在这里结构性失败。
- **两种组合形态**（待实测选）：
  1. **GA 独立验证长臂**：GA 直接搜"红钥→开门→过 boss"的目标骨架，先证明 GA 能到 beam 到不了的门后。
  2. **Hybrid（GA 定骨架 + beam 填细节）**：GA 搜出目标顺序骨架 → 每两个相邻目标之间的段内最优用 beam/`search_quotient` 填（decoder 的 `navigate_to` 换成 beam 段搜）。GA 管"先去哪后去哪"、beam 管"这一段怎么走最省血"。
- 共享一切底座：同一个 `step` 引擎、同一套 `detect_big_items` 目标涌现、同一套区势能/血瓶塔无关口径、同一个 `.h5route` 回放交付。

## 风险与未决问题

1. **`navigate_to` 定向导航器的方向启发可能选错路**。`_toll_dist_from` 是乐观穿门距离，与"真实要先攒钥匙才能开门"有 gap；选错算子会让基因解码得偏。缓解：错路只降这条基因的 fitness、被淘汰，不产非法路线；必要时给 `navigate_to` 加"门控感知"距离（考虑手里钥匙）。**这是实现期最大的不确定点。**
2. **余钥匙的塔无关价值量化未定**。第一版可简化为 `key:* 计数 × 小权重`；进阶用 `door_value.py` 的"门后 pocket 可兑现价值"折算（红钥价值 = 它能解锁的 boss 门后的过 boss 巨值）——但这又把钥匙价值和"门后有什么"耦合，需小心不退回 κ=1（守"拿到才兑现"）。标为第二阶段课题。
3. **fitness 权重标定**。`w_potion / w_key / λ / 通关奖励` 的相对尺度决定 GA 选哪类路线。用现成 route（718 耗尽 vs 0.7 高潜力）当标尺校准，玩家终审。
4. **"门"作为目标的粒度**。一座塔门很多，全部当目标会炸大基因空间。可能只把"门后有未兑现价值的门"（`door_value.py` 已能塔无关识别、`docs/handoff.md` 实测一区真正有价值的门极少）纳入目标集，空袋门不进基因。
5. **`_remaining_items` 跨区扫描扩展**。现仅扫 `current_floor`，GA 要算整区/全塔地上血瓶，需扩范围（小改、保持数据驱动）。
6. **decoder 开销**。population × generations 次全程 `step` 重放，需估算单次 decode 的 step 数 × 总评估次数是否在可接受时间内（`_copy_state` 深拷是主成本；可考虑 `_single_floor_copy` 优化或 fitness 缓存）。
7. **基因里目标的"依赖顺序"**。某目标够不够得到依赖前置目标（先拿钥匙才能开门）；基因顺序不合法时 decoder 选择"跳过"——但这可能让 GA 难学到正确的前置顺序。是否需要"修复算子"（把 pickup_key 自动排到 open_door 之前）待评估，但**绝不写死"红钥在前"这类塔特定规则**（该让 GA 从 fitness 涌现）。
8. **与 `[[feedback-detour-avoid-timing]]` 的关系**：前期属性低时该绕路避战省血、拿盾后再回来打——GA 的目标顺序天然能表达"先绕去拿盾、再回头打前期怪"，可能正好是这条 feedback 的搜索化身。待验证 GA 是否自发涌现这种顺序。
