# 魔塔最优路径求解器

用 Python 搜索算法，求解魔塔通关后**剩余 HP 最大化**的路径。首个目标是 h5mota.com/games/51（经典 50 层），但**架构必须从一开始就为"复用到任意魔塔"设计**：给一座塔的 URL，程序能调取数据、建模其机制、自行搜索最优路线。

## 最重要的原则

- 你的职责是**编写并调试求解程序**，不是自己解塔。**绝不在对话里手动推演、心算或猜测路径**——所有寻路与属性计算都由运行代码完成。
- **始终用简体中文回复。**
- 游戏数据以塔的源码（地图数组 + 事件脚本 + 怪物/道具/门定义）为**唯一事实来源**。禁止凭空编造，禁止靠观察游戏行为推断机制。**遇到数据里没写清、或看不懂的自定义事件，停下来问我或标记为待确认，绝不猜测**——泛化到别的塔时这条更关键，每座塔都可能有任意自定义脚本。
- **★"加 X"= 在【现有所有做法】之上再加 X，不是单独把 X 拎出来做。** 我说"加某个东西"永远是**结合**、不是丢弃之前的做法去单独做一个。例：我说"加 fly"= "字典序 + mult=1 + 之前所有 + fly"，绝不是"只做 fly"。每次扩展前先在心里复述当前已有的全套做法，确认新东西是**叠加**上去的。

## 架构铁律：引擎与塔数据分离

- 把**与具体塔无关的引擎/求解器**和**某座塔特有的数据/机制**彻底分开。`sim/` 和 `solver/` 里**不准出现任何"第10层如何""某怪如何"的硬编码**；这些只能来自 `data/<塔>/`。
- 目标：50 层做完后，换一座塔 = 换一份 `data/`，引擎与求解器一行不改。
- 标准机制（钥匙/门/宝石/商店/战斗）由通用引擎处理；非标准的自定义事件走"适配层"声明，引擎不认识的事件**报错/标记待建模，绝不静默猜测**。

## 战斗模型（每次会话都要记住）

- 战斗完全确定、无随机数，但**损血不是一条简单公式**：怪物带多种「special」属性会修正伤害——先攻、魔攻（无视防御）、坚固（防御下限）、2/3 连击、反击、中毒/衰弱/诅咒（战后）、领域/阻击（按格损血）、夹击（两只同怪之间血减半）、自爆等。
- **权威伤害模型 = 复现游戏引擎自带的伤害函数（getDamage / getDamageInfo），并以引擎算出的损血作为 oracle 逐一校验。绝不凭记忆或常识手写战斗公式。**
- 每个 special 的精确公式、作用顺序、引擎对照表都写在 `data/games51/mechanics.md`，不写在这里——该文档是战斗的事实来源。
- 一般情况下 `你攻 ≤ 怪防` 则打不动该怪（标记不可击杀）；但 special（如魔攻、坚固）会改变这一判断，以 mechanics.md 为准。
- **地形伤害真实存在**（毒地/血网等），按格触发损血，须从源码读出每层地形数据并在模拟器里实现。
- 推论：属性的边际价值是**非线性/阶梯状**的（防御跨过怪攻、攻击跨过连击/坚固等阈值都会让损血突变），搜索绝不能假设"每点属性等价"（优化策略见 docs/solver-design.md）。

## 建图铁律（上一版 bug 的根源）

- 可达性必须**从当前 state 实时计算**：每格是否可通行 = 当前状态的纯函数（看墙 / 门 flag / 钥匙 / 活怪）。**禁止预计算一张静态图反复查。**
- 地图连通性是**动态的**：事件会关/开机关门、移动怪物、撞墙开路。任何改变地图的事件之后，绝不复用旧的可达性结果。
- 范例（50 层第十层埋伏）：踩触发格后通往骷髅队长的内层机关门关闭，须 6 骷髅 + 2 骷髅士兵全部击杀后才开。

## 工程约定

- Python 3.x，遵循 PEP8（4 空格缩进）。
- 模拟器核心是纯函数 `step(state, action) -> new_state`，无副作用，便于单测。
- 目录结构：
  - `data/<塔>/` — 从源码提取的地图、怪物、道具、门、地形、事件（JSON），每条附来源注释。塔特有的一切都在这里。
  - `sim/` — 塔无关的确定性模拟器与战斗逻辑。
  - `solver/` — 塔无关的搜索、剪枝、优化。
  - `extract/` — 从 URL 调取并转换成 `data/` 的提取脚本。
  - `tests/` — 单元测试与存档重放。
- 提取每层数据时，把事件脚本的每条操作（关门/移怪/给奖励/开门/地形等）**逐条翻译**成模拟器状态转移，不要笼统概括。
- 任何从源码读到、或与我确认过的机制和数值，**立即落盘到 `data/` 或 `docs/`**。只活在对话里的结论不算数（会随上下文压缩丢失）。

## 验证与测试

- 模拟器正确性靠两条独立验证：(1) 重放我的通关存档，必须能通关且全程属性吻合；(2) 求解器输出的路径丢回真实引擎重放，必须走通。**引擎只当裁判，不进搜索循环。**
- 每个事件机关都要写单元测试（例：埋伏触发后、8 只未清完时断言队长格不可达；清完后断言可达）。
- 我的存档**只用作正确性校验 + 基准下界**（结果必须不差于它），**绝不让求解器模仿它的走法**。

## 不要做的事

- 不要缓存地图图结构而不在改变地图的事件后让它失效。
- 不要靠玩游戏的观察推断机关规则——去源码里抽脚本。
- 不要在对话里替代码"心算"路径、属性或损血。
- 不要把任何塔特有的数值/机关硬编码进 `sim/` 或 `solver/`。
- 不要把"剑盾优先""加到某阈值"等优化偏好写成死规则——那是搜索该自己算出的结论，不是预设（理由见 docs/solver-design.md）。
